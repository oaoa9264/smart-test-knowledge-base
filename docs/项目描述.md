# 项目描述（当前实现版）

最后更新：2026-03-18（基于当前仓库代码）

## 1. 项目定位

这是一个“测试设计与知识沉淀平台”，核心不是用例台账，而是把测试设计链路结构化：

需求 -> 规则树 -> 用例绑定 -> 覆盖评估 -> 变更影响 -> 回归推荐

并提供 AI 辅助入口（需求文本解析、需求拆解生成与导入）。

## 2. 当前已落地能力（MVP：P0 + P1 + P1+）

### 2.1 项目与需求管理
- 项目 CRUD（`/api/projects`）
- 需求 CRUD（`/api/projects/{project_id}/requirements`）
- 需求来源类型支持：`prd` / `flowchart` / `api_doc`

### 2.2 规则树建模
- 规则节点 CRUD（`/api/rules/nodes`）
- 节点支持类型：`root/condition/branch/action/exception`
- 自动校验父子关系，防止环
- 节点变更后自动重算路径（RulePath）

### 2.3 用例管理
- 用例 CRUD（`/api/testcases`）
- 支持绑定规则节点和规则路径
- 用例状态支持：`active / needs_review / invalidated`

### 2.4 覆盖矩阵
- 项目维度、需求维度覆盖查询（`/api/coverage/...`）
- 输出总节点数、已覆盖节点数、覆盖率
- 输出高风险未覆盖节点与未覆盖路径

### 2.5 变更影响分析
- 规则节点更新/删除后，自动分析受影响用例
- 受影响用例自动标记为 `needs_review`

### 2.6 回归推荐（新增核心能力）
- 接口：`POST /api/reco/regression`
- 支持两种模式：
  - `FULL`：全量目标节点
  - `CHANGE`：只针对变更影响域
- 支持按用例状态过滤，并持久化推荐运行记录（run/result）
- 推荐策略为“风险加权覆盖集”贪心选择，输出：
  - 推荐用例排名
  - 风险覆盖率
  - 剩余高风险缺口

### 2.7 AI 需求解析（轻量）
- 接口：`POST /api/ai/parse`
- 当前实现为 `LLM 优先 + 规则分句兜底` 的草稿节点生成
- 用于快速起草规则树，再人工修订并确认导入

### 2.8 需求拆解（图文 -> 结构化产物）
- 接口：
  - 分析：`POST /api/ai/architecture/analyze`
  - 详情：`GET /api/ai/architecture/{analysis_id}`
  - 导入：`POST /api/ai/architecture/{analysis_id}/import`
- 产物包含：`decision_tree`、`test_plan`、`risk_points`、`test_cases`
- 支持按选项导入正式库（规则节点/风险标注/用例）
- 支持分析引擎标识：
  - `llm`
  - `mock`
  - `mock_fallback`（LLM 失败或结构不合规时降级）

### 2.9 风险收敛闭环（新增）
- 保留原始入口：`POST /api/ai/risks/analyze` 继续负责首轮风险识别，RiskPanel 中原有「分析」按钮未移除
- 新增三阶段收敛动作：
  - `POST /api/requirements/{requirement_id}/snapshots/review`：生成评审阶段有效需求快照
  - `POST /api/ai/risks/predev-analyze`：基于最新快照和规则树执行开发前分析
  - `POST /api/ai/risks/prerelease-audit`：在提测前做阻塞风险、重开风险和已解决风险审计
- 风险列表已升级为连续账本，`RiskItem` 支持 `analysis_stage`、`validity`、`origin_snapshot_id`、`last_seen_snapshot_id`，不再按每轮分析整体重置
- 需求输入与有效需求快照已经持久化：
  - `POST /api/requirements/{requirement_id}/inputs`
  - `GET /api/requirements/{requirement_id}/inputs`
  - `GET /api/requirements/{requirement_id}/snapshots`
  - `GET /api/requirements/{requirement_id}/snapshots/latest`
- 产品知识检索已升级为 evidence-backed 方式：先查 `evidence_blocks`，不足时再回退原始文档分块
- Product Docs 页面支持“手工补录知识”，即使没有 `risk_item_id` 也能直接生成待审核的文档更新建议

## 3. 技术架构

- 前端：React + TypeScript + Vite + Ant Design + React Flow
- 后端：FastAPI + SQLAlchemy
- 数据库：默认 SQLite（可按文档切换 Docker Postgres）
- 文件：流程图上传后存储于 `backend/uploads/architecture/`

## 4. 当前价值

- 把“需求-规则-用例”关系从文档文本变为可计算结构
- 把覆盖率和高风险缺口量化展示
- 把规则变更后的回归范围从“拍脑袋”变成可追踪推荐
- 把架构评审阶段输出（判断树、风险点、用例矩阵）可落库沉淀
- 把风险识别从“一次性列表”升级为可跨评审 / 开发前 / 提测前持续演进的账本
- 把产品知识从原始文档分块升级为“证据块 + 手工补录 + 文档更新建议”的闭环

## 5. 与传统用例管理平台的差异

- 核心对象是“规则节点/路径”，不是“目录+标签”
- 强调设计期建模、覆盖推导、影响分析、回归推荐
- AI 能力直接服务“测试设计产物生成与导入”，不是仅做问答

## 6. 当前边界（尚未覆盖）

- 尚未覆盖测试执行编排、缺陷流转、测试报告闭环
- AI 需求解析仍是轻量规则解析，不是通用 LLM 推理
- 权限体系、审计、复杂协作流程仍是后续增强方向
