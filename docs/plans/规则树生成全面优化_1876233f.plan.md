# 规则树生成全面优化方案

## 问题诊断

当前同一输入多次生成规则树，节点数在 30-44 间波动，内容"飘"。根因是四层问题叠加：

1. **Temperature = 1.0**（最大随机性）
2. **Prompt 缺乏粒度约束**（不知道拆到多细、何时合并）
3. **两阶段管道放大方差**（Stage 1 自然语言不稳定 -> Stage 2 继承放大）
4. **无后处理兜底**（重复节点、过多/过少节点没有代码层面的清理）

---

## 第一层：LLM 参数调优

**文件**: [backend/.env](backend/.env), [backend/.env.example](backend/.env.example)

- `LLM_TEMPERATURE`: 1.0 -> **0.15**（结构化输出场景的最佳实践区间 0.1-0.2）
- 新增 `LLM_SEED=42` 环境变量，在 payload 中传入 `seed` 参数，进一步提升可复现性

**文件**: [backend/app/services/base_llm_client.py](backend/app/services/base_llm_client.py)

- `__init` 新增 `seed` 参数，从 `LLM_SEED` 读取
- `_create_completion` 的 payload 中加入 `seed` 字段（仅当 seed 不为 None 时）

**文件**: [backend/app/services/zhipu_client.py](backend/app/services/zhipu_client.py), [backend/app/services/openai_client.py](backend/app/services/openai_client.py)

- 构造函数传入 `seed` 参数

## 第二层：Prompt 体系重设计

**文件**: [backend/app/services/prompts/architecture.py](backend/app/services/prompts/architecture.py)

### Stage 1 - VISION_SYSTEM_PROMPT（全部重写）

设计原则：**模板化 + 定长 + 反幻觉**

```
角色：测试架构分析师
任务：将流程图+描述转写为结构化的架构理解文本
输出格式：纯自然语言，禁止 JSON/表格/markdown

## 严格按以下 3 个区块输出（总字数 800-1200 字）：

【主流程】（200-300 字）
用"步骤1 -> 步骤2 -> ..."的链式写法，写出从入口到结束的主干路径。
- 只写 3-8 个关键步骤
- 遇到判断点用"若X则A；否则B"标注分支

【条件分支表】（300-500 字）
逐条列出所有条件判断，格式固定为：
  条件N：<可判定的布尔描述>
    -> 满足时：<可观察的系统行为>
    -> 不满足时：<可观察的系统行为>
要求：
- 条件必须可判定（是/否、等于、不为空、状态=X）
- 行为必须可观察（展示/隐藏、可编辑/只读、默认值=X、提示文案=X）
- 相同行为在不同分支下出现时，只写一次并注明适用范围

【异常与边界】（200-400 字）
列出异常场景，格式固定为：
  异常N：<触发条件> -> <系统表现> -> <恢复方式/标注"未明确">
```

核心变化：

- 从 5 个章节精简到 **3 个区块**（合并了"高风险"和"测试关注点"到条件分支和异常中，因为这两个章节在 Stage 1 容易产生空泛内容，真正有价值的信息已经在条件和异常里了）
- 每个区块有**字数范围**，总量控制在 800-1200 字（配合后面把截断阈值提升）
- **固定格式模板**（"条件N：... -> 满足时：... -> 不满足时：..."），减少模型的自由发挥空间
- 明确写了"相同行为只写一次"，从源头减少下游冗余

### Stage 1 - VISION_USER_TEMPLATE（微调）

增加一句强调：`严格遵守 system prompt 中的区块格式和字数要求，不要自由发挥。`

### Stage 2 - GENERATE_SYSTEM_PROMPT（全部重写）

在现有 JSON 格式约束基础上，**新增三大块约束**：

**A. 节点粒度规则**（解决节点数波动）：

```
节点粒度标准：
- 一个 condition 节点 = 一个可判定的布尔决策点（是/否问题）
- 一个 action 节点 = 一个可观察的系统行为（一句话能说清）
- 一个 branch 节点 = 一个条件的结果路径（"是"或"否"）
- 一个 exception 节点 = 一个异常/边界处理

合并规则：
- 如果同一个 action/exception 在多个平行分支下重复出现，
  提取为它们共同父节点的子节点，只保留一份
- 连续的无条件步骤（A必然接B接C）合并为一个 action

拆分规则：
- 如果一个步骤包含独立的判断逻辑，必须拆为 condition + branch
- 如果不同产品/类型有不同行为，每种行为独立为 branch
```

**B. 节点数量指引**：

```
目标节点数：20-35 个（典型业务流程）
- < 15 个说明粒度太粗，检查是否遗漏了条件分支
- > 40 个说明粒度太细或存在重复，检查是否可以合并
```

**C. 精简的 few-shot 示例**（约 10 个节点的小例子，展示理想的树结构和粒度标准，避免用户实际业务的例子以防过拟合）

### Stage 2 - GENERATE_USER_TEMPLATE（增强）

新增一段"检查清单"：

```
生成后请自检：
1. 是否有内容高度相似的节点？ -> 合并
2. 节点总数是否在 20-35 区间？ -> 调整粒度
3. 每个 condition 是否都有对应的 branch？ -> 补全
4. 是否有孤立节点（无 parent_id 且非 root）？ -> 修正
```

## 第三层：管道代码增强

**文件**: [backend/app/services/architecture_analyzer.py](backend/app/services/architecture_analyzer.py)

### 3.1 提升截断阈值

```python
# 当前（第 113 行）
return (understanding or description or "")[:2000]

# 改为
return (understanding or description or "")[:4000]
```

配合 Stage 1 prompt 的 800-1200 字约束，4000 字符足够容纳完整输出，不会被随机截断。

### 3.2 新增节点去重后处理

在 `_validate_and_fallback` 之前（或之后），加一步 `_deduplicate_nodes`：

逻辑：

- 遍历所有节点，找出 content 相似度 > 80% 且 type 相同的节点对
- 如果它们的 parent 是同级兄弟（共同祖先），保留其中一个，将另一个的子节点重新挂载
- 相似度计算用简单的字符级 SequenceMatcher（标准库 difflib），不引入额外依赖

### 3.3 节点数量校验日志

在验证阶段加一个 warning log：

- 节点数 < 15：warning "节点数过少，可能粒度太粗"
- 节点数 > 40：warning "节点数过多，可能存在冗余"

不做硬拦截（不影响现有流程），只记录日志供后续分析。

## 预期效果

- **稳定性**：同一输入多次调用，节点数波动从 30-44（47% 方差）收敛到 25-30（~15% 方差）
- **质量**：消除重复节点（如"是否已选择套餐"不再出现 3 次），树更精炼
- **可控性**：通过 temperature/seed/节点数指引三重约束，输出可预期
