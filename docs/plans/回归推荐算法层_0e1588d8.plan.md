# 回归推荐算法层实现计划

## 方案可行性结论

方案思路正确，与现有代码结构高度契合。核心依据：

- `case_rule_node_assoc` / `case_rule_path_assoc` 已提供 Cover(c) 数据基础
- `RuleNode.risk_level` 可直接映射为风险权重
- 单父树 `parent_id` 天然支持祖先/子树遍历
- `derive_rule_paths` 已实现 DFS 路径推导

## Sprint 1：最小可用（核心算法 + FULL 模式 + 解释输出）

### 1.1 修复 DFS 环保护（前置必做）

在 [rule_engine.py](backend/app/services/rule_engine.py) 的 `dfs` 函数中增加 `visited` 集合，防止 parent_id 形成环时栈溢出。同时在节点保存时增加环检测（沿 parent 链向上走，看是否遇到自己）。

### 1.2 新增数据表（2 张）

在 [entities.py](backend/app/models/entities.py) 中新增：

- `**reco_run**` -- 推荐运行记录
  - `id` (Integer PK), `requirement_id` (FK), `mode` (Enum: FULL/CHANGE), `k` (Integer), `input_changed_node_ids` (Text/JSON, nullable), `total_target_risk` (Float), `covered_risk` (Float), `coverage_ratio` (Float), `created_at` (DateTime)
- `**reco_result**` -- 推荐结果明细
  - `id` (Integer PK), `run_id` (FK -> reco_run.id), `rank` (Integer), `case_id` (FK -> test_cases.id), `gain_risk` (Float), `gain_node_ids` (Text/JSON), `top_contributors` (Text/JSON), `why_selected` (String)

Sprint 1 不建 `node_risk_score` 和 `case_cover_cache`，风险权重和覆盖集合都在内存中实时计算。

### 1.3 风险权重计算服务

新建 `backend/app/services/risk_scorer.py`：

```python
def compute_risk_scores(nodes, tree_stats, config) -> dict[str, float]:
    """
    对 requirement 下所有活跃节点计算 risk_score。
    risk_score = a * type_weight + b * complexity + c * change_freq + d * uncovered_bonus

    - type_weight: critical=4, high=3, medium=2, low=1
    - complexity: children_count / max_children (归一化)
    - change_freq: node.version / max_version (归一化)
    - uncovered_bonus: 1.0 if uncovered else 0.0
    """
```

- `tree_stats` 从节点列表计算：每个节点的子节点数、子树大小、深度
- `config` 存储系数 a/b/c/d，默认值 `{"a": 3.0, "b": 1.0, "c": 1.5, "d": 2.0}`
- `uncovered` 信息从现有 `build_coverage_matrix` 的结果中提取

### 1.4 覆盖集合计算

新建 `backend/app/services/cover_set.py`：

```python
def compute_cover_sets(cases, path_map) -> dict[int, set[str]]:
    """
    计算每个用例的覆盖节点集合。
    Cover(c) = bound_node_ids ∪ {n for p in bound_paths for n in path_node_sequence}
    """
```

- `path_map`: path_id -> node_id 列表，从 RulePath.node_sequence 解析
- 输入直接从 DB 的 relationship 获取，无需额外表

### 1.5 贪心推荐算法（核心）

新建 `backend/app/services/recommender.py`：

```python
def recommend_regression_set(
    universe: set[str],           # 需要覆盖的节点集合
    risk_weights: dict[str, float], # 节点风险权重
    cover_sets: dict[int, set[str]], # case_id -> 覆盖节点集合
    k: int,                        # 回归K
    cost_mode: str = "UNIT"        # 成本模式
) -> RecommendationResult:
    """
    贪心迭代：每轮选 gain/cost 最大的用例。
    返回排序结果列表 + 每步解释。
    """
```

- 每轮迭代：遍历候选用例，计算 `new_cover = Cover(c) ∩ universe - Covered`，`gain = sum(risk_weights[n] for n in new_cover)`
- 停止条件：达到 K 条，或 gain <= 0
- 输出：`RecommendationResult` 包含 summary + cases 列表 + remaining_gaps

### 1.8 编排层（串联 API）

`POST /api/reco/regression` 的处理流程：

```
1. 查询 requirement 下所有活跃节点 (status != deleted)
2. 查询所有路径 (RulePath)
3. 查询符合 case_filter 的用例 (关联该 requirement)
4. 计算 tree_stats (子节点数、深度等)
5. 计算 risk_weights (risk_scorer)
6. 计算 cover_sets (cover_set)
7. 确定 universe:
   - FULL 模式: 全部活跃节点
   - CHANGE 模式: compute_impact_domain(changed_node_ids)
8. 调用 recommend_regression_set(universe, risk_weights, cover_sets, k)
9. 落库 reco_run + reco_result
10. 返回 RecoResponse
```

## Sprint 2：变更回归模式

### 2.1 影响域计算

新建 `backend/app/services/impact_domain.py`：

```python
def compute_impact_domain(changed_node_ids, nodes) -> set[str]:
    """
    计算变更影响域：
    - 变更节点本身
    - 祖先链 (ancestor chain，沿 parent_id 向上)
    - 子树 (subtree，DFS 向下)
    """
```

在树结构中，这是 O(n) 操作。

### 2.2 CHANGE 模式权重加成

- 影响域内节点 risk_weight 乘以 change_boost 系数（默认 1.5）
- 影响域外节点保持原始权重

### 2.3 升级现有影响分析

可选：将 [impact.py](backend/app/services/impact.py) 的 `analyze_impact` 也升级为使用 `compute_impact_domain`，使影响分析从"节点交集"升级到"树域传播"。

## Sprint 3：性能与成本优化

### 3.1 成本模式扩展

- `cost_mode=TIME`：用例执行耗时作为成本（需要给 TestCase 加 `estimated_duration` 字段或从外部获取）
- 贪心公式变为 `gain / duration`

### 3.2 缓存层

- 按需增加 `case_cover_cache` 表，基于 requirement 规则树版本 hash 做增量刷新
- 风险权重缓存（`node_risk_score` 表），避免重复计算

### 3.3 RuleNode 加 `updated_at`

为 RuleNode 新增 `updated_at` 字段，支持基于时间窗口的变更频率计算。
